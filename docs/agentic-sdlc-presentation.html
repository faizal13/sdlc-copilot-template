We're Not Just Using AI — We've Wired It Into Our Process

Spec-driven development means writing better documentation before coding. We went further: our AI doesn't just read specs, it creates them, acts on them, reviews the output, and learns from each sprint.

⚠ Spec-Driven Development

  • → Engineers write detailed specs manually before coding
  • → AI used as an autocomplete tool at code-write time
  • → Spec quality depends on individual engineer experience
  • → Story refinement is a slow manual meeting exercise
  • → No audit trail of AI usage or decision reasoning
  • → Knowledge stays in people's heads, not the system

✓ What We Do: Agentic SDLC

  • ✓ AI reads business epics & generates technical specs automatically
  • ✓ Specialised agents handle planning, coding, review, and telemetry
  • ✓ Consistent quality enforced by agent instructions across the team
  • ✓ Story refinement runs in minutes, not hours
  • ✓ Every AI action logged with session, agent, and outcome
  • ✓ Domain knowledge codified and shared across every engineer
The Pipeline

From Business Requirement to Reviewed Code

Each stage of our delivery pipeline has a dedicated AI agent. They hand off to each other — the output of one becomes the input of the next.

📋

Business Epic Arrives in Azure DevOps @story-refiner

The BA writes the epic in plain business language. Our Story Refiner agent reads the entire epic tree — every feature and every story — analyses technical impact across all microservices, builds a dependency graph, and generates a phased execution plan. It also posts clarification questions back on ADO stories when business requirements are ambiguous.

Technical child stories created in ADO Dependency graph Phased execution plan Gap report
📐

API Contract Designed Upfront @api-architect

Before any engineer writes a single method, our API Architect agent reads the execution plan and solution design docs and produces industry-standard OpenAPI 3.1 specifications — as if a senior API architect had designed them. It enforces RFC 9457 error shapes, cursor pagination, read/write model separation, and strict schema contracts. The spec becomes the binding contract between frontend, backend, and QA.

OpenAPI 3.1 per-service specs Common error schemas (RFC 9457) Pagination & audit models Contract handoff to dev agents
🔍

Story Analysed Before a Line of Code @story-analyzer

Before any development begins, the Story Analyzer reads the ADO story alongside the solution design, existing codebase, and domain context. It validates the story is complete, unambiguous, and technically feasible — flagging anything that would waste engineering time mid-sprint.

Readiness verdict Risk flags Service impact map
📝

Implementation Plan Generated @task-planner

The Task Planner breaks the story into a sequenced, file-by-file implementation plan — entities, migrations, services, controllers, DTOs, tests — with exact class names and method signatures derived from the actual codebase. Engineers know exactly what to build before they start.

Ordered task list File-level instructions Test strategy
⚡

Code Written by AI, Guided by Domain Rules @rakbank-backend-dev-agent

Our development agent writes production-ready Spring Boot code following our exact architecture standards — package structures, naming conventions, security patterns, and integration rules. It knows our domain, our tech stack, and our non-negotiables. It doesn't just generate code; it generates our code.

Production-ready code Unit tests Liquibase migrations
🔎

Code Reviewed Against Our Standards @local-reviewer

Before any human reviews a PR, an AI reviewer checks the code against our architecture rules, security requirements, test coverage, and acceptance criteria. It leaves structured comments on the PR — categorised by severity — so human review time is focused on business logic, not style or missing null checks.

Structured PR comments Security flags AC coverage verdict
📊

Every Sprint Measured and Learned From @telemetry-collector

Every agent action is logged — what was asked, which agent responded, how long it took, and what was produced. At the end of each sprint the Telemetry Collector produces an AI usage report: time saved, stories completed with AI assistance, quality scores. The system learns sprint over sprint.

Sprint AI usage report Time-saved estimate Quality trends

Design Principles

Why It Works at Scale

The system is built on principles that make AI behaviour predictable, auditable, and continuously improving.

🎯 Specialised, Not General

Each agent has a single, well-defined job and deep context for that job. A reviewer only reviews. A planner only plans. Focused agents produce better, more consistent output than one general-purpose assistant.

📚 Domain Knowledge is a First-Class Asset

Our banking domain rules, architecture decisions, security requirements, and coding standards are codified in context files that every agent reads before acting. New engineers onboard faster. Senior knowledge doesn't walk out the door.

🔗 ADO is the Single Source of Truth

Agents read from and write to ADO directly. Technical stories, clarification comments, and execution plans live alongside BA stories — no context switching, no separate AI workspace.

📈 Every Sprint Makes the System Smarter

The Instinct Extractor learns from code review outcomes and codifies team instincts back into the agent context. The system gets better with each sprint without any manual tuning.

🔒 Human Oversight by Design

Agents plan and assist — humans decide. No agent can merge a PR, close a story, or deploy to production. AI accelerates every step; engineers remain accountable for every decision.

📋 Full Audit Trail

Every AI session is logged with timestamp, agent, prompt, and outcome. Every ADO write is tagged with the agent that created it. Compliance and quality reviews have a complete record.

📐 API-First by Design

The API contract is defined before any code is written. Our API Architect agent produces OpenAPI 3.1 specs that dev agents, reviewers, and QA all reference. Frontend and backend can build in parallel against the same contract — no integration surprises at merge time.
